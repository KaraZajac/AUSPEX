# AUSPEX QA Spot-Check 2026-05-26

## Methodology

- **Sample size**: 50 events
  - 25 from the 2025–2026 cycle (recent additions, post-Bybit through
    January 2026)
  - 25 from the older corpus (2009–2024)
- **Selection**: deterministic stride sampling across the sorted file
  list within each bucket (every Nth file), so the sample is
  reproducible. Not formally random but is independent of authorship
  order.
- **Verifier**: Claude agent, 2026-05-26
- **Method**: for each event, read the YAML; verify internal
  consistency (dates, attribution language vs. cited source, doctrine
  reasoning, FK targets); cross-reference the attribution / source
  claim against the Pass 1 source-URL findings; sanity-check
  quantified impact and malware-family detail in summaries against
  open-source memory. Primary-source HTTP fetches were performed
  selectively (Trend Micro, Mandiant, Treasury, NSArchive, DefenseScoop,
  CCDCOE, IranIntl, WaterISAC, Talos) for items where the source URL
  itself was in doubt from Pass 1. A full fetch of every cited source
  was not feasible inside the time budget; uncertain calls below are
  labelled accordingly.

The QA pass deliberately tries to be lenient on doctrine link
reasoning (which is editorial-judgment territory by design) and
strict on date / attribution / source-pointer accuracy (which is
factual).

## Headline

- **34 events** fully verified, no discrepancies of consequence.
- **9 events** with minor discrepancies (date drift inside the source
  publication window, sub-cluster naming choices, attribution
  language slightly stronger than the source supports, occasional
  attesting-source vs. counter-explanation symmetry issue).
- **6 events** with material discrepancies (wrong actor slug
  attributed, source-citation pointing at a stub that does not
  match its claimed date / publisher, manufactured specificity on
  attribution).
- **1 event** where verification not possible (source dead-link or
  paywalled, no recoverable mirror).

## Material discrepancies

### 1. `events/2025/04/fbi-salt-typhoon-bounty.yaml` — wrong actor slug on attribution row

The FBI Rewards for Justice $10M bounty is for information on **Salt
Typhoon operators** (PRC MSS-aligned). The event YAML attributes the
bounty action to `actor_id: us/cybercom/cnmf-actor` (a US actor slug
used elsewhere as a placeholder for US offensive-cyber attribution).
That is the wrong directionality: this row should attribute to the
PRC actor whose operators the bounty is *targeting*, not the US
issuer. The issuer ("FBI Rewards for Justice") is already correctly
in `attributing_org`.

Recommended correction: change `actor_id` to `cn/mss/salt-typhoon`
(matching the actor slug already used in
`2025-06/salt-typhoon-canada-telecom.yaml` and the August
AA25-239A advisory event), and either remove the `actor_id` line
or surface the US-issuer status by leaving `actor_id: null` with a
`service_id` / `attributing_org` framing instead. The placeholder
comment in the YAML itself flags this as known-to-be-wrong.

Citation: see CISA AA25-239A 2025-08 attribution and OFAC's January
2025 Sichuan Juxinhe designation; both consistently use the
`cn/mss/salt-typhoon` cluster framing.

### 2. `events/2025/02/ofac-iran-cyber-officials-designation.yaml` — source stub date does not match a real Treasury press release

The event claims an OFAC tranche on **2025-02-12** against IRGC-IO
Cyber-Electronic Command officials. Pass 1 verification turned up
no Treasury press release on that date. The most plausible referent
is **jy2072 (2024-02-02)**, which designated six IRGC-CEC officials
(Hamid Reza Lashgarian and five subordinates) in response to the
Aliquippa Unitronics PLC defacement. If the AUSPEX event is the
correct concept, its `start_date` / `disclosure_date` should be
2024-02-02 and the cited source URL should be `https://home.treasury.gov/news/press-releases/jy2072`.

If a distinct February 2025 IRGC-IO designation does exist (a
sb-series action, or under a separate jy-series number), it was
not surfaced by web search and would need to be located in OFAC's
recent-actions index.

Citation: Treasury press release jy2072, 2024-02-02.

### 3. `events/2025/05/apt36-pahalgam-operation-sindoor-tempo.yaml` — citation date mismatch

The event cites `cyfirma/2025-05-15_apt36-pahalgam-tempo` for an
attribution dated 2025-05-15. The canonical Cyfirma report
"Firewalls and Frontlines: The India-Pakistan Cyber Battlefield
Crisis" was published 2025-06-05 (post-Sindoor retrospective). I
updated the source stub's `published_on` to 2025-06-05 with a note;
the event's `disclosure_date` and `attribution_date` may need to be
re-set to 2025-06-05 to match. The slug retains the 2025-05-15
midpoint date by editorial convention.

Citation: https://www.cyfirma.com/research/firewalls-and-frontlines-the-india-pakistan-cyber-battlefield-crisis/

### 4. `events/2021/06/boston-childrens-hospital-thwarted.yaml` — manufactured specificity on attribution

Wray's June 2022 Boston College Cyber Conference statement
characterized the intrusion attempt as "Iran government-sponsored."
He did **not** name the Najee Technology / Afkar System contractor
cluster. The event attributes to `actor_id: ir/irgc/najee-technology`
with `attributing_org_confidence: high` and `auspex_assessment:
concur`. The methodology page §"What we don't do" explicitly
prohibits upgrading service-level FBI language to a specific cluster.

Recommended correction: set `actor_id: null` and `service_id: ir/irgc`
(or whichever IRGC-IO-equivalent service slug exists). The YAML's
own `notes:` field acknowledges Wray "did not name a specific
contractor" but the row's `actor_id` still asserts the cluster.

Citation: FBI Director Christopher A. Wray, "The Director's Remarks
at the Boston Conference on Cyber Security 2022," 2022-06-01.

### 5. `events/2023/12/alphv-fbi-disruption.yaml` — actor/target slug confusion

Two related issues:
- Attribution row uses `actor_id: us/cybercom/cnmf-actor`, which is
  the same US-issuer placeholder problem flagged in (1) above. The
  FBI-DOJ disruption action is being attributed to the US
  Cybercom CNMF actor — but ALPHV is a Russia-tolerated
  ransomware brand; the row should either attribute to the ALPHV
  operators (with `actor_id` pointing at the ALPHV / BlackCat
  cluster slug if one exists) or use `actor_id: null` with the
  service / attributing org framing.
- `targets[0].target_id: ru/proxies/alphv` looks like an **actor**
  slug used as a `target_id`. Targets should resolve to
  `targets/orgs/...` or sector slugs. ALPHV's leak-site
  infrastructure is the actual disruption target — would be better
  modelled as `infra/alphv-leak-site` or similar.

Recommended correction: review the slug conventions for "we
disrupted an actor's infrastructure" events.

Citation: FBI press release, "U.S. Department of Justice Disrupts
ALPHV/Blackcat Ransomware Variant," 2023-12-19.

### 6. `events/2025/06/iran-israel-12-day-war-cyber-surge.yaml` — doctrine link pillar/reasoning mismatch

The doctrine link to `ir/post-soleimani-retaliation/electoral-discord`
carries reasoning about "targeting of US critical infrastructure
following the 22 June Fordow / Natanz / Isfahan strikes" — the
*reasoning* describes a CII-pre-positioning / retaliatory-disruption
behavior, but the *pillar* slug ("electoral-discord") is about a
different operational mode (US-election-targeting / IRGC influence
operations). Either the pillar slug is wrong or the reasoning is
wrong; they don't match.

Recommended correction: either move the doctrine link to the right
pillar under `ir/post-soleimani-retaliation` (likely a
critical-infrastructure-retaliation pillar) or revise the reasoning
to genuinely describe election-discord behavior during the 12-day
war.
