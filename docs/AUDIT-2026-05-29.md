## A. Methodological validity threats — require a decision

### editorial-campaign-id-label-leakage  `[high]`
*Editorial campaign_id is a high-weight feature that is near-1:1 with an actor for a large fraction of tagged events; LOO does not neutralize it because the campaign→actor association persists in the N-1 training set*

- **Where:** `site/src/utils/attribution.ts:261 (campaign feature), :657 (default weight 4.0 — second-highest), :706-714 (fires only when actor seen in same campaign in training); analogous doctrine weight 2.5 (doctrine-prediction.ts:173)`
- **Class:** methodology-validity · finder call: ambiguous · no number moves
- **Evidence:** Reproduced via tsx: of 153 actor-labeled events carrying a campaign_id, 63 (41.2%) belong to a campaign that maps to exactly one distinct actor corpus-wide; 18 of 33 distinct campaigns are single-actor. The campaign feature in scoreActor only fires when the candidate actor already appears in that campaign in the N-1 training set (line 710), so for any multi-event single-actor campaign the held-out event's campaign_id deterministically points at the true actor with weight 4.0*IDF*log(1+count). campaign_id is an editorial field whose value is often assigned because analysts already knew the attribution.
- **Claim impact:** A meaningful share of the attribution headline's top-1 hits on campaign-tagged events are driven by an editorial field that encodes the label. Defensible reading: a publicly-reported campaign linkage is legitimate 'visible-fabric' evidence available at prediction time. Adversarial reading: it is label leakage through analyst-assigned metadata that the engine would not have for a genuinely novel event. The 4.0 weight makes it decisive when it fires.
- **Recommendation:** Quantify the headline with the editorial campaign feature ablated (set features.campaign=null in LOO) and report both numbers, framing campaign_id as an optional 'known-linkage' input. A full LOO ablation was set up but not completed here; it should be run to state the exact pp contribution before defense.
- **Verifier verdict:** `confirmed-validity-threat` (real=True, defensible-design=True, confidence=high)
- **Strongest refutation:** A publicly-reported campaign linkage is genuine evidence that an analyst possesses at prediction time for a KNOWN operation — the author deliberately frames campaign_id as the "visible-fabric" signal (attribution.ts:56-57) and consciously weights the editorial campaign (4.0) above the algorithmically-inferred one (1.5), showing it is a considered design choice, not an oversight. The 41 lost hits are real public attributions, so the engine is reproducing correct, defensible analyst reasoning. One can therefore argue the headline is the engine's performance WITH all available evidence, and campaign is simply the most discriminating legitimate feature (analogous to a named operator). This refutation is real but partial: it does not rescue the ~24 strictly single-actor-campaign cases where campaign_id is tautological with the label, and it does not excuse the methodology page never disclosing the feature or reporting the ablated number, so the headline silently overstates performance on genuinely novel (untagged) events by ~8pp.

So one engine yields two published top-1 numbers (59.4% vs 54.9%) differing only by the denominator convention. Counting the 41 as misses: 298/543=54.9% top-1 (-4.5pp), 394/543=72.6% top-3 (-5.9pp), MRR 0.700*502/543=0.647.

If null-dropping is kept, report the dropped count and rate prominently and reconcile the two pages. The stacked-eval note (lines 238-247) mischaracterizes the gap as an MRR top-K artifact; it is actually the null-denominator drop and should be corrected.
- **Verifier verdict:** `confirmed-validity-threat` (real=True, defensible-design=False, confidence=high)

The attribution-eval page also literally prints 'n=502 labeled events scored', so the denominator is not entirely hidden. HOWEVER this refutation fails to rescue the PUBLISHED number because: (a) the README headline has no n and no caveat and is labeled flatly 'Current accuracy (leave-one-out)'; (b) standard IR practice scores an unretrieved relevant item as RR=0 / miss, not exclusion; and (c) the same engine is published at 54.9% on the stacked-eval page ('Plain NB') versus 59.4% here — an internal contradiction the project cannot defend, and one the author's own commit cad3b7f already resolved in favor of null=miss ('silently skipped ... inflated both engines').

### inferred-campaign-loo-contamination  `[low]`
*inferredCampaign feature is clustered once over the FULL corpus and cached on the atlas singleton, so during LOO a held-out event shares an inferred-cluster id with the training events it is scored against*

- **Where:** `site/src/utils/atlas.ts:1189 (inferredCampaignByEvent = inferCampaigns(_atlas)); consumed at attribution.ts:262, scored at 719-725; LOO at attribution-eval.ts:80-84 never recomputes clusters`
- **Class:** methodology-validity · finder call: ambiguous · moves a published number
- **Evidence:** inferCampaigns iterates all 658 events (inferred-campaigns.ts:108) and the result is stored on the singleton. attribution-eval LOO builds profiles on training=N-1 (78-80) but extractFeatures(heldOut,a) (83) reads a.inferredCampaignByEvent.get(heldOut.id) — the full-corpus cluster id — and training events in the same cluster accumulate inferredCampaigns counts (attribution.ts:433), so scoreActor (719-725) rewards an actor whose other in-cluster events carry its label. Bounded: 35/658 events carry an inferred-campaign id across 10 clusters (all >=2 events), all 35 labeled non-editorial, ~7% of 502 scored. Clustering excludes actor/doctrine/pillar labels (inferred-campaigns.ts:12-13, edges only via target/malware/lineage), so this is label-propagation-via-structural-similarity, not direct leakage — but the held-out event still helps form the cluster used to score it.
- **Claim impact:** README attribution LOO 59.4/78.5/MRR0.700 (and OOD-entropy/stacked which share the feature). Bounded to <=35 events. Honest readings: (1) mild optimistic bias because clusters were formed with the held-out event present; (2) defensible because clustering uses no labels, so a per-fold recompute would largely reproduce the same structural transfer.
- **Recommendation:** For LOO/temporal eval, recompute inferred campaigns per training fold (exclude the held-out event), or zero the held-out event's inferredCampaign in the eval feature vector and report the delta as a stated protocol.
- **Verifier verdict:** `confirmed-validity-threat` (real=True, defensible-design=False, confidence=high)
- **Strongest refutation:** The corpus-wide effect is negligible: my exact reproduction (restricting the held-out loop to the only 35 events that can possibly differ, since every other event's inferredCampaign feature is null in all conditions) shows that switching from the published full-corpus clustering to an honest per-fold recompute changes the rank of exactly 2 of 33 scored IC-carrying events — flipping one top-1 (patchwork-us-think-tanks 1→2) and one within-top-3 drop (sidecopy 2→3). On the ~502-event scored set that is top-1 59.4%→~59.2% (-0.20 pp), top-3 unchanged at 78.5%, and MRR 0.700→0.699 (-0.0013).
